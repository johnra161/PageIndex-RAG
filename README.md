# PageIndex RAG

A FastAPI-based Retrieval-Augmented Generation (RAG) system built around hierarchical PageIndex retrieval and long-context LLM answering.

This project allows users to upload PDF documents, automatically index them into hierarchical tree structures using PageIndex, and query them using multiple retrieval strategies while tracking latency and token usage.

---

# Features

- Upload and index PDF documents
- SHA-256 document hashing for deduplication
- Automatic index reuse for previously uploaded documents
- Hierarchical PageIndex retrieval (tree traversal RAG)
- Long-context retrieval (entire document in one prompt if context window allows)
- Side-by-side comparison between retrieval methods
- Token usage tracking
- Latency measurement
- Query result caching
- Background document processing
- Interactive hierarchical tree visualization
- Real-time traversal visualization during retrieval
- Expandable/collapsible document nodes
- Retrieval explainability through node state coloring
- FastAPI backend with lightweight frontend

---

# Interactive Tree Visualization

The indexed PageIndex tree is visualized directly in the frontend UI.

When a document is uploaded:
- the generated hierarchical tree is displayed automatically
- if the document was previously indexed, the cached tree is loaded instantly

Users can:
- expand and collapse nodes
- navigate through the document hierarchy interactively
- inspect the document structure generated by PageIndex

This allows users to visually understand how the document was segmented and indexed.

---

# Query Traversal Visualization

During hierarchical retrieval, the traversal process is visualized live on the tree.

Each node changes color depending on how the retrieval system interacted with it:

| State | Meaning |
|---|---|
| Grey | Node was not visited |
| Dark Grey | Node was visited but skipped as irrelevant |
| Blue | Node was explored further |
| Green | Node was ultimately used in answer synthesis |

This makes the retrieval process interpretable and allows users to observe:
- how the LLM navigates the document hierarchy
- which branches were ignored
- which sections contributed to the final answer

The visualization acts as a debugging and explainability tool for hierarchical RAG traversal.

# Retrieval Modes

## 1. Hierarchical (PageIndex RAG)

Uses PageIndex-generated document trees and recursively traverses relevant sections using LLM-guided navigation.

### Benefits
- Lower token usage
- Faster retrieval on large documents
- Structured exploration of document hierarchy
- Interpretable retrieval path visualization

---

## 2. Long Context

If the document fits within the model’s context window, the entire document is provided directly to the LLM.

### Benefits
- Simpler retrieval pipeline
- Useful baseline for comparison

### Limitations
- Higher token usage
- Slower on large documents
- Context window constraints

---

## 3. Compare Both

Runs both retrieval strategies and returns:

- Both generated answers
- Token usage for each method
- Latency for each method
- Relative efficiency comparison

This makes it possible to compare hierarchical retrieval against direct long-context prompting on the same document and query.

---

# Caching

Query results are cached on disk.

If the same user asks the same question on the same document again:
- cached results are returned instantly
- no additional LLM calls are made
- no additional tokens are consumed

Caching is performed separately for:
- hierarchical retrieval
- long-context retrieval

# Architecture

```text
Upload PDF
    ↓
SHA-256 Hashing
    ↓
Check Existing Index
    ↓
If New:
    Run PageIndex
    Build Tree
    Store Tree JSON
Else:
    Reuse Existing Tree
    ↓
Display Interactive Tree
    ↓
User Query
    ↓
Choose Retrieval Mode
    ↓
Tree Traversal / Long Context
    ↓
Generate Answer
    ↓
Cache Result
```

# Notes

- Documents are uniquely identified using SHA-256 hashes.
- Previously indexed documents are automatically reused.
- Large documents may skip long-context mode if they exceed the context window limit.
- Query caching has a configurable TTL.
- Hierarchical traversal uses bounded breadth exploration with adaptive stopping.
- Query results include token usage and latency metrics for retrieval analysis.
